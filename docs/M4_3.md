# M4.3 — Power Curve Visualizer

Closed: 2026-05-01

## What it does

Renders ally vs enemy team power curves over 3 game-length buckets. Two modes:
- **team** : two superimposed lines (ally blue, enemy red) for the comp aggregate.
- **lanes** : five mini-curves (one per role) stacked, for granular reading.

Plus a `dominantBucket()` helper to identify the bucket where the ally team
has the largest advantage — used by Risk Markers (M4.4) and the prototype's
power-window callout.

## Public API

```typescript
import { calculatePowerCurve, dominantBucket } from '$lib/strategic/powerCurveCalculator';
import PowerCurveVisualizer from '$lib/components/PowerCurveVisualizer.svelte';

const curve = calculatePowerCurve(allyKeys, enemyKeys, dataset);
const window = dominantBucket(curve); // 'early' | 'mid' | 'late' | null
```

```svelte
<PowerCurveVisualizer {allyComp} {enemyComp} {dataset} mode="team" />
```

## Bucket mapping (D7 with documented deviation)

The spec D7 asked for:
- early = 0-15 min
- mid = 15-25 min
- late = 25+ min

But the upstream DraftGap dataset's `statsByTime` array uses 25/30/35/40 minute
boundaries (verified by inspection 2026-05-01). The visualizer maps the 5
dataset buckets to 3 visualizer buckets:

| Visualizer bucket | Dataset indices | Real game length |
|---|---|---|
| `early` | [0] | <25 min |
| `mid` | [1, 2] (combined) | 25-35 min |
| `late` | [3, 4] (combined) | 35+ min |

The labels reflect the actual boundaries. See `docs/unresolved.md` for
context.

## Calculation method

Per `D11` of the M4 sprint spec:
- Per champion, compute its bucket winrate as `wins / games` (combining
  indices when needed).
- Team bucket winrate = arithmetic mean across champions in the comp.
- Champions with no data in a bucket are excluded from that bucket's average.
- If all champions miss data in a bucket, the visualizer renders a gap.

The arithmetic-mean choice (instead of weighted-by-games) is intentional: it
prevents one heavily-played champion from dominating the team curve.

## Visualizer details

- SVG inline, viewBox-responsive, no canvas.
- Y axis fixed [0.30, 0.70] (typical winrate spread; clamped on extremes).
- Reference dashed line at 50%.
- Ally = `text-blue-500`, enemy = `text-red-500`.
- Data points are circles at each bucket; absent data → gap in the path.
- Delta callouts in pp (percentage points) at the bottom.

## Tests

12 cases covering:
- 3 buckets in correct order
- Early-strong vs late-strong inversion
- Bucket aggregation math (indices 1+2, 3+4)
- Null-entry exclusion
- Missing champion graceful degradation
- Unknown key handling
- Arithmetic-mean (not weighted)
- Lane variant returns 5×3=15 points
- `dominantBucket` correctness
- Live-data smoke (skipped if dataset sample absent)

## Limits

- The visualizer assumes role assignment is correct (index 0=top, …, 4=support).
  Comp inputs that scramble roles will produce nonsense numbers.
- The component doesn't handle live updates beyond Svelte 5's `$derived` —
  changing the dataset prop forces a re-render.
- Lanes mode crowds at narrow widths (<400px). Acceptable for the prototype;
  responsive collapse is a M8 polish item.
