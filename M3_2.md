## Two adapter-level fixes that landed in this milestone

Both surfaced because LCK has a different format than LFL. Neither is
scope creep — both are necessary for the LCK adapter to produce the
same quality of output as LFL.

### 1. `resolveSeriesDate` (BO5 series-date fallback)

LFL Spring is BO1, so each row in
`tournament-matchlist/<slug>/` is one game and the `gameId → date`
map is exhaustive. **LCK Cup is BO5**: the matchlist lists one row
per *series* (mapped to the first game of the series, e.g. 74754).
The other games of the series have adjacent IDs (74755–74758) and
are absent from the matchlist.

First T1 scrape after wiring LCK: 7/23 drafts had `playedAt`. After
the fallback: 23/23.

`resolveSeriesDate(map, gameId)` :
- exact match → that ISO date,
- otherwise pick the largest map ID ≤ `gameId` within a 10-ID cap,
- return null beyond cap.

The 10-ID cap is safe vs. cross-series leakage: LCK Cup BO5 series
span ≤5 IDs adjacent, and the next series starts ≥60 IDs later
(other matches of the day fall in between in gol.gg's global ID
space). Five unit tests cover exact-match / fallback / beyond-cap /
cross-series safety / null / invalid id.

### 2. Champion mapping: `RenataGlasc` slug alias

Riot id = `Renata`, display = `Renata Glasc`, gol.gg image filename
= `RenataGlasc.png`. My `bySlug` index only mapped Riot's `id` →
key, so the slugified display name was missed. 3 unmapped warnings
on T1.

Fix: `buildChampionIndex` now also indexes
`normalizeChampionSlug(champ.name)` as a slug alias. No-op when id
and name slugify to the same string (most champions). Catches the
multi-word display-name case generally — a future-proofing for
champs that follow the same pattern.

## Pool-match ratios — interpretation

The 65–69% ratio is consistent across the three leagues, which
matters for M3.3:

- It's not a LFL-specific artifact. Pro rosters everywhere have
  ~30% of pick attributions that fall to the dataset-primary-role
  fallback (substitute games, off-meta first-time picks).
- LCK is *slightly* lower than LFL (66.2 vs 68.5) — likely because
  LCK has more games per team (BO5) which means more opportunity for
  one-off picks not yet in the player's pool.
- The number is stable enough that a calibration pass treating all
  leagues uniformly is reasonable. We don't need league-specific
  knobs for M3.3.

## Findings worth carrying forward

- **246-game corpus is N≥200, the threshold I flagged in M2 as
  "what would actually validate M2"**. M3.3 calibration should now
  produce statistically meaningful results.
- **Date coverage is 100% across all three leagues** thanks to the
  series-fallback. Walk-forward eval (M3.3) has the dates it needs.
- **gol.gg's User-Agent filtering**: short UAs (`DraftLab/0.1`) get
  404 on tournament-stats and tournament-matchlist while team-list
  works. Full UA `DraftLab/0.1 (+url)` works everywhere. Already in
  the prod transport — just noted here for posterity.
- **Roster turnover since the M3.2 spec was written**: LCK's KDF
  Freecs is now HANJIN BRION; LEC's MAD Lions and BDS were replaced
  by Movistar KOI, Natus Vincere, and Shifters. The team list pulls
  current rosters automatically — no spec update needed.

## Limitations

- **`tournament-stats` page** is not used by any of the four scrapers
  (team-stats, team-draft, team-matchlist via team-list, player). I
  skipped auditing it; the four flow paths are sufficient.
- **No fixture for `lck-matchlist.html` and `lec-matchlist.html`
  was used in regression tests for the series fallback** because the
  unit tests already cover that with synthetic data. The fixtures
  are committed and exercised by the date-coverage parser tests
  (≥80% assertion).
- **LPL still a stub** — option in the dropdown is disabled. Slug
  format unverified, deferred to M3.5+.

## Questions for M3.3

1. **Recalibration with N=246** vs. **walk-forward eval with N=246**:
   the first re-runs the M2 grid search on a much bigger corpus to
   see if the optimal `defaultPriorGames / sidePriorGames /
   cheeseAttenuationFactor` shift. The second uses
   `playedAt` to reconstruct pool-as-of-date and remove the
   temporal leak. Order matters because walk-forward bumps the noise
   floor — we'd want both, but which first?
2. **Mapping date → patch**: M3.2 has `playedAt` per draft but the
   dataset is fixed at one patch ("30 days"). Walk-forward against
   varying-patch drafts would need a `patch → champion-data`
   snapshot for each historical patch. That's a real data pipeline
   item, not just an analyzer tweak.
3. **Substitute-player modeling**: the 31–34% dataset-fallback
   attributions across all three leagues look like a real
   signal (pros do play subs and one-offs). The cleanest first cut
   I sketched in BACKLOG was "if pick wasn't matched to roster pool,
   set `comfortMode='unavailable'` so the analyzer falls back to
   baseline". Worth doing before another calibration pass to avoid
   the analyzer fabricating a pool signal that doesn't exist.
