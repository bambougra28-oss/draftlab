# M3.1 — Critical Tightening

Three corrections, in order, behind a single commit.

## 1. parseRecentDrafts role attribution (was wrong in production)

**Before.** `parseRecentDrafts` assigned roles by pick-order index
(`[Top, JG, Mid, Bot, Sup][i]`) on the 5 picks in HTML order. Pro
drafts pick by phase-order, not by role, so this was wrong for
basically every game. The calibration runner (`tests/m2-calibration.
test.ts`) papered over this by re-attributing roles from rosters
before computing anything — duplication.

**After.** `attributeRoles` and `championPrimaryRole` are exported from
`src/lib/pro/golgg.ts`. `parseRecentDrafts` accepts an optional
`{ players, dataset }` and uses these helpers when both are passed:

1. **Pool match.** Each pick is looked up in every roster member's
   pool; a unique match assigns that player's role.
2. **Dataset fallback.** Picks with no match get the champion's
   primary role from the dataset (the role with the most games).
3. **Leftover sweep.** Anything still unfilled gets the remaining
   roles in order.

`fetchTeam` now passes `players + dataset` to `parseRecentDrafts`, so
production `ProTeam.recentDrafts` arrives with correct roles.

The calibration runner had `championPrimaryRole` and `attributeRoles`
deleted (≈85 lines gone). It now reads `team.recentDrafts[].picks`
directly. A small `countPoolMatches` helper kept the
"% pool-attributed" diagnostic for the calibration report.

When `players` or `dataset` is absent (existing fixture-only parser
tests), the parser falls back to the old pick-order behavior. This
keeps the `parseRecentDrafts(html, index, name)` two-arg call site
working without churning every test.

### Tests added

- `parseRecentDrafts × role attribution (pick-swap test)` — 4 cases:
  - drafts have unique roles per pick (no duplicates) on all 8 ZYB games
  - ≥75% have a complete 5-pick × 5-role bijection
  - **Olaf in ZYB game 77034 attributes to TOP, not pick-order TOP**
    (Olaf was the 5th pick in HTML order; pick-order would have stamped
    him as Support. Pool-match correctly puts him under Wao = TOP.)
  - **Neeko attributes to SUP, not pick-order BOT** (Neeko was the
    4th pick in HTML order; Riippp's pool contains Neeko = SUP.)
  - explicit fallback test: without `players + dataset`, the parser
    keeps the old pick-order behavior

## 2. Game date scraping

**Source.** Dates do not live on the team-stats, team-matchlist, or
team-draft pages. They're on the **tournament-wide** matchlist:
`https://gol.gg/tournament/tournament-matchlist/<slug>/`. One fetch
returns dates for every game in the split.

**Format.** gol.gg renders date-only locale-naive strings
(`2026-04-23`). For walk-forward filtering we need a fully-qualified
ISO 8601 datetime, so we anchor to **midnight UTC** of the displayed
day. Trade-off: a game played on the evening of 2026-04-23 (CEST)
and one in the early hours of 2026-04-24 may both round to the same
UTC midnight — fine for "before this date" filters, lossy for
sub-daily ordering. Documented in code.

**Plumbing.**

- `urlTournamentMatchList(slug)` — URL builder.
- `parseTournamentMatchList(html)` — returns `Map<gameId, isoUtc>`.
  Locates the table by checking `headers.includes('Game') &&
  headers.includes('Date')`, then scans cells from the right looking
  for a `YYYY-MM-DD` literal. Layout-drift resilient.
- `fetchTeam` calls `fetchWithCache(urlTournamentMatchList(...))`
  best-effort and passes the resulting map into `parseRecentDrafts`.
  If the matchlist fetch fails, the team scrape continues with
  `playedAt = null` and a warning entry.
- `RecentDraft.playedAt: string | null` — added to the type. The old
  `date` field is kept as a mirror for any caller that filtered on it.

### Tests added

- `parseTournamentMatchList (LFL Spring 2026)` — 3 cases:
  - returns ≥30 game→date entries
  - every value is a valid `YYYY-MM-DDT00:00:00.000Z`
  - **game 77034 → 2026-04-23T00:00:00.000Z** (verified by hand
    against gol.gg)
- `parseRecentDrafts × gameDates` — 3 cases:
  - drafts get `playedAt` set when their gameId is in the map
  - missing gameId → `playedAt = null` (the "absent" case)
  - legacy `date` field mirrors `playedAt`
- Live test (`golgg.live.test.ts`, env-gated):
  **assertion: ≥80% of ZYB drafts have non-null `playedAt`**.
  Actual on 2026-05-01: **8/8 (100%)**.
